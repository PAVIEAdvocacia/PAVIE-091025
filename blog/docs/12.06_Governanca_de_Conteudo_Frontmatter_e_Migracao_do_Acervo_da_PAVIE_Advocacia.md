# 12.06 — Governança de Conteúdo, Frontmatter e Migração do Acervo da PAVIE Advocacia

## 1. Finalidade

Este documento fixa a camada de governança executável entre a arquitetura editorial da PAVIE e a implementação em Astro/Decap. Seu objetivo é impedir que o CMS produza conteúdo tecnicamente publicável, mas semanticamente desgovernado.

Ele regula:

1. coleções mínimas;
2. frontmatter canônico;
3. enums controlados;
4. esquema de validação;
5. fluxo de migração do acervo legado;
6. critérios de arquivamento, redirecionamento e saneamento.

---

## 2. Princípios

### 2.1 Fonte única de verdade

O frontmatter não deve repetir em texto livre o que pode ser resolvido por código, enum ou mapa central.

### 2.2 Poucas chaves, alto valor

Cada chave deve cumprir função editorial, técnica, analítica ou de governança real.

### 2.3 Campo livre é exceção

Quanto mais sensível a informação, mais controlado deve ser o valor.

### 2.4 Migração é reclassificação, não simples cópia

Acervo legado não deve ser despejado no novo sistema sem curadoria.

---

## 3. Coleções mínimas recomendadas

## 3.1 `posts`

Responsável pelos artigos do blog.

## 3.2 `areas`

Responsável pelas páginas de área correspondentes às categorias canônicas.

## 3.3 `authors`

Responsável por perfis públicos de autoria.

## 3.4 `settings` ou `taxonomy`

Opcional na fase inicial. Só criar se melhorar governança de fato.

---

## 4. Frontmatter canônico para posts

## 4.1 Campos obrigatórios

| Chave | Tipo | Regra |
|---|---|---|
| `title` | string | título público do post |
| `seoTitle` | string | pode repetir `title`, mas deve existir |
| `description` | string | meta description editorialmente útil |
| `excerpt` | string | resumo curto para cards/listagens |
| `slug` | string | único, estável, em minúsculas e sem data obrigatória |
| `publishDate` | date | data de publicação |
| `authorId` | enum | inicialmente `fabio-pavie` |
| `categoryCode` | enum | `CAT-01` a `CAT-07` |
| `contentType` | enum | tipo editorial autorizado |
| `readerStage` | enum | estágio do leitor |
| `ctaType` | enum | CTA principal do post |
| `ctaTarget` | string | slug da área, URL interna ou rota canônica |
| `draft` | boolean | controle editorial |
| `noindex` | boolean | controle de indexação |
| `featured` | boolean | destaque em listagens |
| `legalReview` | enum | status de revisão jurídica |
| `editorialReview` | enum | status de revisão editorial |

## 4.2 Campos opcionais controlados

| Chave | Tipo | Regra |
|---|---|---|
| `updatedDate` | date | usar quando houver revisão material |
| `coverImage` | string | caminho estável do asset |
| `coverAlt` | string | obrigatório se houver imagem |
| `tags` | string[] | secundárias e controladas |
| `relatedPosts` | string[] | slugs relacionados |
| `relatedAreas` | string[] | slugs das páginas de área |
| `canonicalURL` | string | usar quando houver canonical externa |
| `schemaType` | enum | padrão `Article` ou `BlogPosting` |
| `legacyId` | string | vínculo com acervo anterior |
| `redirectFrom` | string[] | slugs antigos |
| `migrationStatus` | enum | estado do saneamento |

## 7. Exemplo de schema para Astro content collections

```ts
import { defineCollection, z } from "astro:content";

const categoryCodes = [
  "CAT-01",
  "CAT-02",
  "CAT-03",
  "CAT-04",
  "CAT-05",
  "CAT-06",
  "CAT-07",
] as const;

const posts = defineCollection({
  type: "content",
  schema: z.object({
    title: z.string().min(12),
    seoTitle: z.string().min(12),
    description: z.string().min(50).max(180),
    excerpt: z.string().min(50).max(240),
    slug: z.string().regex(/^[a-z0-9-]+$/),
    publishDate: z.coerce.date(),
    updatedDate: z.coerce.date().optional(),
    authorId: z.literal("fabio-pavie"),
    categoryCode: z.enum(categoryCodes),
    contentType: z.enum(["cornerstone", "guide", "spoke", "faq", "checklist", "case-note", "institutional"]),
    readerStage: z.enum(["discover", "clarify", "compare", "decide"]),
    ctaType: z.enum(["area", "contact", "article-series", "document-review"]),
    ctaTarget: z.string(),
    draft: z.boolean().default(false),
    noindex: z.boolean().default(false),
    featured: z.boolean().default(false),
    legalReview: z.enum(["pending", "reviewed", "needs-adjustment"]),
    editorialReview: z.enum(["pending", "reviewed", "needs-adjustment"]),
    coverImage: z.string().optional(),
    coverAlt: z.string().optional(),
    tags: z.array(z.string()).default([]),
    relatedPosts: z.array(z.string()).default([]),
    relatedAreas: z.array(z.string()).default([]),
    canonicalURL: z.string().url().optional(),
    schemaType: z.enum(["Article", "BlogPosting"]).default("BlogPosting"),
    legacyId: z.string().optional(),
    redirectFrom: z.array(z.string()).default([]),
    migrationStatus: z.enum(["native", "migrated", "revised", "archived", "redirected"]).default("native"),
  }).superRefine((data, ctx) => {
    if (data.coverImage && !data.coverAlt) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: "coverAlt é obrigatório quando houver coverImage.",
        path: ["coverAlt"],
      });
    }
  }),
});

export const collections = { posts };
```

## 9. Regra de naming e slugs

1. slug curto, sem artigo desnecessário, mas com contexto suficiente;
2. não repetir a categoria inteira dentro do slug se o título já a pressupõe;
3. evitar anos e datas no slug, salvo motivo estratégico;
4. um slug publicado só deve mudar se houver necessidade real e redirecionamento.

---

## 10. Política de migração do acervo legado

## 10.1 Estados possíveis do acervo

Cada item legado deve receber um dos seguintes destinos:

- `migrar-como-esta` — raro;
- `migrar-com-ajuste` — regra geral;
- `fundir` — quando houver duplicidade temática;
- `arquivar` — quando o texto não sustentar a nova arquitetura;
- `redirecionar` — quando o valor residual estiver na URL histórica;
- `reaproveitar-como-spoke` — quando o núcleo for útil, mas a forma antiga não.

## 10.2 Checklist de saneamento por item legado

1. mapear categoria canônica;
2. definir tipo editorial;
3. revisar título, descrição e excerpt;
4. normalizar slug;
5. marcar revisão jurídica e editorial;
6. verificar CTA;
7. decidir redirect;
8. registrar `legacyId`, se houver.

## 10.3 Regra de poda

Se o conteúdo legado:

- conflita com a taxonomia;
- exagera áreas não prioritárias;
- usa tom incompatível com a matriz pública;
- entrega conteúdo desordenado ou frágil;

ele não deve ser migrado como ativo vivo.

---

## 11. Política de campos derivados

Não armazenar manualmente, salvo necessidade:

- tempo de leitura;
- label público da categoria;
- breadcrumbs;
- textos de CTA padronizados;
- URL canônica interna derivável;
- schema gerado automaticamente.

Esses dados devem preferencialmente ser derivados por mapa central, utilitário ou componente.

---

## 12. Critérios de aceitação

O sistema de frontmatter e coleções só deve ser considerado apto quando:

1. o CMS impede ou reduz fortemente erro de classificação;
2. o schema do Astro recusa entradas inválidas;
3. os campos sensíveis são enum ou seleção controlada;
4. a migração do legado não reintroduz a dispersão anterior;
5. a estrutura é simples o bastante para manutenção contínua.

---

## 13. Cláusula final

O frontmatter da PAVIE não serve para “enriquecer” o Markdown com burocracia. Serve para fazer o conteúdo obedecer ao sistema.
