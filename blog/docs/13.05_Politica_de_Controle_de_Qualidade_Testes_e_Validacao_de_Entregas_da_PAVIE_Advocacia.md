# 13.05 — Política de Controle de Qualidade, Testes e Validação de Entregas da PAVIE Advocacia

## 1. Finalidade

Este documento fixa o regime mínimo de QA do ecossistema web da PAVIE para impedir que mudanças tecnicamente concluídas sejam tratadas como entregas aptas sem validação estrutural, editorial e institucional.

---

## 2. Princípio central

Na PAVIE, “funcionou no navegador” não equivale a “está apto para publicar”.

A validação deve percorrer, cumulativamente:

1. **integridade técnica**;
2. **integridade editorial**;
3. **integridade institucional**;
4. **integridade de conversão ética**.

---

## 3. Portões de qualidade

## 3.1 Gate A — pré-commit ou pré-merge técnico

Verificar, no mínimo:

- build do Astro;
- validação do schema das collections;
- typecheck, se houver TypeScript;
- lint, se configurado;
- erro de import, rota ou componente;
- render mínimo das rotas críticas.

## 3.2 Gate B — QA editorial e de superfície

Verificar, no mínimo:

- categoria canônica correta;
- CTA compatível com a matriz pública;
- ausência de ampliação indevida do portfólio;
- correspondência entre post, área e autor;
- clareza de heading e legibilidade.

## 3.3 Gate C — pré-publicação

Verificar, no mínimo:

- título, description, canonical e metadata;
- links internos;
- breadcrumb;
- alt de imagens;
- versão mobile;
- ausência de noindex indevido;
- revisão jurídica e editorial concluídas.

---

## 4. Critérios mínimos por tipo de entrega

## 4.1 Página de post

Checklist:

- [ ] frontmatter válido
- [ ] `categoryCode` presente e correto
- [ ] `contentType` coerente
- [ ] `ctaType` e `ctaTarget` presentes
- [ ] autor definido
- [ ] heading structure clara
- [ ] página de área vinculada
- [ ] relacionados preenchidos ou derivados corretamente
- [ ] metadata renderizada
- [ ] leitura confortável em mobile

## 4.2 Página de área

Checklist:

- [ ] nome canônico correto
- [ ] resumo compatível com a Matriz Canônica
- [ ] sem ampliação de área
- [ ] CTA institucional sóbrio
- [ ] lista de posts relacionados funcional
- [ ] schema e metadata coerentes

## 4.3 Página `/sobre` e bio do autor

Checklist:

- [ ] sem autoengrandecimento
- [ ] sem fatos não documentados
- [ ] sem competir com `/areas/`
- [ ] coerente com a matriz pública
- [ ] links internos corretos

## 4.4 Home

Checklist:

- [ ] não opera como catálogo exaustivo
- [ ] valor institucional claro
- [ ] navegação limpa
- [ ] CTA contido
- [ ] áreas canônicas apresentadas sem inflação

---

## 5. Validações automatizáveis recomendadas

Sempre que possível, automatizar:

1. schema de frontmatter;
2. checagem de slug duplicado;
3. presença de `coverAlt` quando houver imagem;
4. correspondência entre `categoryCode` e mapa central;
5. build de todas as rotas críticas;
6. link interno quebrado;
7. páginas órfãs, quando viável.

---

## 6. Validações humanas irrenunciáveis

Não delegar integralmente a automação:

1. adequação do tom institucional;
2. excesso de promessa ou conversão indevida;
3. coerência com a matriz de superfícies públicas;
4. risco de ampliação implícita do portfólio;
5. pertinência do CTA;
6. qualidade argumentativa do texto.

---

## 7. Tratamento de não conformidades

## 7.1 Severidade

### Bloqueadora
Impede merge ou publicação.

Exemplos:
- quebra de build;
- schema inválido;
- categoria indevida;
- bio com informação não comprovada;
- CTA eticamente inadequado.

### Alta
Permite correção imediata antes de publicar.

Exemplos:
- metadata fraca;
- link quebrado;
- ausência de relacionados;
- mobile ruim.

### Média
Não impede o merge técnico, mas impede publicação final sem ajuste.

Exemplos:
- inconsistência visual pontual;
- resumo abaixo do padrão;
- interlinking pobre.

### Baixa
Pode entrar em backlog.

Exemplos:
- refinamento menor de microcopy;
- ajuste discreto de espaçamento;
- melhoria incremental de preview.

---

## 8. Evidências mínimas de aprovação

Cada entrega relevante deve deixar ao menos uma trilha curta contendo:

- escopo alterado;
- arquivos principais tocados;
- risco residual conhecido;
- pendências assumidas;
- resultado dos gates principais.

---

## 9. Checklist mestre de publicação

Antes de publicar qualquer rota ou conteúdo relevante, verificar:

- [ ] build concluído
- [ ] schema válido
- [ ] taxonomia canônica respeitada
- [ ] matriz pública respeitada
- [ ] CTA compatível
- [ ] mobile conferido
- [ ] links conferidos
- [ ] metadata conferida
- [ ] revisão jurídica concluída
- [ ] revisão editorial concluída

---

## 10. Critério de aceite final

Uma entrega só deve ser considerada apta quando:

1. não produz regressão estrutural;
2. não fragiliza a identidade pública;
3. não dificulta a manutenção;
4. pode ser operada por humano e por CMS;
5. sustenta crescimento posterior com baixo retrabalho.

---

## 11. Cláusula final

Controle de qualidade, na PAVIE, não é acabamento cosmético. É barreira contra erosão institucional.
